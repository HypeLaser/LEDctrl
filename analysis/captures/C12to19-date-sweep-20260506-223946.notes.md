## Side find — `0f 32` between row 1 and row 2

After first row separator `0d` came `0f 32` before next token. Not seen in single-token messages. Hypothesis: row-prefix font/colour reset block (matches A-set multi-row format where each new row reinjects formatting).
- `0f` = "format change" directive
- `0x32` = font code (matches `30` = ASCII '0' / "default font" pattern)
- Editor only emits this on the *second* row to override default — first row inherits from line 0x070 prefix.

This isn't blocking — not part of the clock-token spec — but log it for the multi-row encoder.
