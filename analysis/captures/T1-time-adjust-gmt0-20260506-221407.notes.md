# T1 — Time Adjustment "Adjust" button (GMT+0)

Capture: `T1-time-adjust-gmt0-20260506-221407.pcap`
Date: 2026-05-06 22:14 BST
Editor input: dialog showed "2026-05-06 22:12  TimeZone GMT+0.00", button = "Adjust"
Wall at click: ~22:14:43 BST

## Wire sequence

Three exchanges:

### [1] 0x0501 READ TIME — request
```
seq 24
arglen 0000
```
Sign response (8B payload):
```
26 20 05 06 22 14 03 0c
yy yy mm dd hh mm DOW flag
2026 May 06  22:14  Wed=3  0x0c
```
Year is little-endian BCD: `26 20` reads as 2026.
Sign returns **its own local wall time** (BST 22:14, since we still had old TZ-home).
DOW=3 with Sun=0 anchor.

### [2] 0x0502 SET TIME — write (no seconds)
```
seq 25
arglen 0200 (2 words = 8B)
payload: 26 20 05 06 21 14 04 0c
         yy yy mm dd hh mm DOW flag
         2026 May 06 21:14 Wed=4 0x0c
```
Editor sent **GMT (21:14)**, not laptop wall (22:14).
DOW=4 with Sun=1 anchor (CHANGED from read response, see below).
0x0c trailer matches read response — protocol-version constant.

### [3] 0x0504 SET TIME+TZ — write (with seconds + TZ)
```
seq 26
arglen 0300 (3 words = 12B)
payload: 26 20 05 06 21 14 43 04 0c 00 00 00
         yy yy mm dd hh mm SS DOW flag <tz 4B>
         2026 May 06 21:14:43 Wed=4 0x0c GMT+0
```
Same time/date as [2] plus seconds=0x43=67? No — wall was 22:14:43, seconds=43 dec = 0x2B not 0x43.
Wait: 0x43 = 67 decimal. But 43 BCD = 43 decimal. So it IS BCD. Seconds=43 ✓.
TZ slot = `00 00 00 00` = GMT+0 ✓.

## Findings

| Opcode | Purpose | Payload bytes | Notes |
|---|---|---|---|
| 0x05:0x01 | READ time | 0 (req), 8 (resp) | Already in SigmaClient — confirm |
| 0x05:0x02 | SET time | 8 | Already in SigmaClient — confirm; no seconds |
| 0x05:0x04 | SET time+TZ | 12 | **NEW**, adds seconds + 4B TZ |

### Encoding

- All time fields **BCD** (e.g. 0x14 = 14 dec, not 0x14 = 20 dec).
- Year = LE BCD: bytes `XX YY` → year `YYXX` (2026 = `26 20`).
- Hour = sent in GMT (21 not 22). Sign owns local-TZ math.
- DOW conventions differ between read response (Sun=0, Wed=3) and write (Sun=1, Wed=4). Mystery.
  - Hypothesis: read is "real" sign-side, write is editor's reformat. Sign accepts either.
  - Test: send DOW=3 in our SigmaClient setSignTime, see if sign accepts.
- 0x0c byte: fixed protocol-version flag. Always present.
- TZ slot: 4 bytes, GMT+0 = all zero. Need T1b capture with non-zero TZ to map encoding (signed minutes? hours? offset enum?).

## Resolves earlier mystery

Memory note `messagemaker_clock_offset.md` documented the +1hr double-offset.
Root cause clear now: our setSignTime sends laptop **local** (BST=GMT+1) not GMT.
Editor uses 0x0504 with explicit GMT-converted time + GMT+0 TZ slot. Sign then re-applies its
own TZ-home setting for display.
**Fix in code**: convert to GMT before BCD-encoding hour byte, set sign's TZ-home explicitly via 0x0504.
