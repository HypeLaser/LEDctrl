## NMG body — per-row encoding pattern

```
07 30                                 text-start
4c 49 4e 45 20 4f 4e 45               "LINE ONE"
0d                                    row separator
4c 49 4e 45 20 54 57 4f               "LINE TWO"
0d 04                                 row sep + footer
```

**Critical**: only ONE row-style block (`31 0e 32 30 30 30 32 1f 30 1e 31 …`) at the top, applied to ALL rows. Subsequent rows are pure text after a `0d` separator. Per-row offset/font/size/colour blocks ONLY appear when fonts/styles differ between rows (mixed-font case from prior `editor-marquee-continuous` capture).

This simplifies our `renderFittedBytes` / `renderMarqueeBytes`: emit one style block, then text rows separated by `0d`.

### Diff vs A2-WON (auto-wrap)

A2-WON inserted `0d 20` (with space) at the wrap point. A4-WON emits clean `0d` between explicit rows, no leading-space artefact. Editor's auto-wrap quirk is a leading-space pad — explicit rows do not have it.

A2-WON and A4-WON both render 2 rows, both wrap ON, but slot1 = 8224 vs 22240. **Slot1 depends on text content, not just structure.**

CRC sweep results (X25, CRC32, Adler32, byte-sum, word-sum LE) — none match slot1 values across all 4 captures. Slot1 is NOT a content checksum.

Working hypothesis: slot1 = **Editor-computed total display duration in some unit** (not pure ms — values too large for 2s pause × N rows). Could be ticks at 2.97 ms each, or related to scroll-pixel budget.

## Critical bug-site confirmation

`SigmaClient.swift:1325-1330` per-entry format:
```swift
[length: 2 bytes][flags 0x0101: 2 bytes][filename: 12 bytes]
```

Vendor wire format observed:
```
[slot1: 2 bytes][slot2 = NMG length: 2 bytes][filename: 8 bytes]
```

**Our code emits NMG length in slot1 and 0x0101 in slot2.** Firmware reads NMG body length from slot2 → reads `0x0101 = 257` bytes regardless of true NMG length. For shorter NMGs the firmware reads garbage past the file; for longer NMGs it reads truncated content. Both produce inconsistent gaps. Marquee-gap root cause confirmed.

Fix sequence (after enough A/K/C/D captures to lock slot1):
1. Slot order: slot1 (timing) → slot2 (length).
2. Filename pad to 8 bytes (not 12).
3. Compute or template slot1 properly.
