# C10 — Three separate inserts (Hour + Min + Sec)

Capture: `C10-three-separate-20260506-223520.pcap`
Date: 2026-05-06 22:36 BST
Editor: Inserted Hour, Min, Sec presets back-to-back
Sign rendered: **"223615"** — no colons, confirms wire

## Token region

```
07 30 [0b 2c 0b 2d 0b 2e] 0d 04
       └H──┘└M──┘└S──┘
```

Body length 222 (C4 Hour-only baseline = 218). Δ = +4 = 2 extra 2-byte tokens. Tokens concatenate cleanly with no separator bytes.

## Findings

- **Multiple clock tokens compose by simple concatenation** — no parent grouping, no field count.
- Each `0b XX` token is independent and self-describing.
- For human-readable HH:MM:SS the *editor must inject literal `:` text bytes between tokens*. User's three-insert action did **not** include colons, so wire carried no colons → sign rendered concatenated digits.
- Define Time (C8 OPT=0x8e) is the **only way** to get protocol-level HH:MM:SS with auto-colons. Three-insert is a manual fallback that requires literal colon text.

## Implication for SigmaProgram emitter

Two API styles:

```swift
// Style 1: Define Time single token (preferred — sign auto-formats)
emitClockToken(.defineTime(twelveHour: false, showSec: true, staticColon: true))
// → 0b 34 2f 8e

// Style 2: Compose from preset tokens + literal text
emitClockToken(.hour) + ":" + emitClockToken(.min) + ":" + emitClockToken(.sec)
// → 0b 2c 3a 0b 2d 3a 0b 2e
```

Style 1 is the "right" way; Style 2 is for fancy layouts (e.g. mixing time with arbitrary text).
