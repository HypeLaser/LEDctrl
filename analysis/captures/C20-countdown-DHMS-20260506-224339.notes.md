## Findings

### Countdown is NOT clock-token format
- **Master directive byte = 0x18** (vs 0x0b for clock).
- After `07 30` prefix: `18 16` then `0a 33` (could be sub-version: 0x33 = '3', maybe ASCII "3"?).
- Editor emits a **literal printf-style format string** (`%d%h%m%s` as ASCII bytes) rather than a per-field enum.
- Fields:
  - `%d` = Day
  - `%h` = Hour
  - `%m` = Minute
  - `%s` = Second
- Sign-side parser substitutes computed counter values into the format string.

### Target time encoding (4 bytes вҖ” TBD)
`a7 5c 00 60` little-endian = `0x60005CA7` = 1610891431
- **Not unix seconds** (1610891431 = 2021-01-17, doesn't match tomorrow noon)
- **Not BCD** (`a7` and `5c` are non-BCD)
- **Not IEEE float of unix seconds**
- **Not seconds-from-2000 epoch**

Hypothesis: encoded delta or sign-internal epoch. **Capture C20b with different target** (e.g. 5 min from now) to map binary form by diffing.

### Mystery ASCII chunk `00101000`
8 ASCII zeros/ones immediately after target bytes. Possibilities:
- Bitfield encoded as text (8 flag chars: count-down=0 or 1, etc.)
- Multiplier "1.0" zero-padded to fixed width as string ("00010100"? doesn't match exactly)
- Reserved/zeroed format options

Unclear. Diff against C20b/C20c will reveal which bits move.

### Multi-row collapse
User inserted 4 separate items (one per row), but Editor produced single row with 4 inline placeholders. Final wire has **2 rows** total: row1 = `%d%h%m%s`, row2 = empty (just `0f 32` prefix then `0d`). Suggests Editor merges adjacent counter inserts into one format string when they share the same counter spec.
