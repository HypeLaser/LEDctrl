## High-Risk Lessons

Treat native `System Set` upload as a deliberate recovery-risk operation until read-back/backup support exists.

A bad `CONFIG.SYS` upload once made the sign think it was a `144 x 96` panel, leaving the display blank. Recovery worked through USB serial by sending a patched known-good config.

Recovery file:

```text
/Users/alexscott/Projects/MessageMaker/LEDPanelRescue/build/recovery-80x7-192.168.11.6-CONFIG.SYS
```

Important patched offsets in the recovery config:

- `0x02`: width `50 00` = 80
- `0x04`: height `07 00` = 7
- `0x24`: IP `06 0B A8 C0` = `192.168.11.6` little-endian
- `0x4c`: newline width `50 00`
- `0x4e`: newline height `07 00`
- `0x88`: gateway `01 0B A8 C0` = `192.168.11.1`
- `0x8c`: mask `00 FF FF FF` = `255.255.255.0`

The native app now prepares a Sigma Play-style `SysInfoFile`, not the earlier recovery `CONFIG.SYS`, and validates that the source template is `80 x 7` before writing a patched file. It can write that file to the sign, but only mapped fields should be editable; do not add unmapped Sigma Play options without one-change captures.

Do not send dense inline colour changes. An early native experiment inserted colour-control tokens before many characters for diagonal palette output. That likely wedged the sign. The safe vendor method uses a single global `1C xx` colour/palette code in the NMG header.

## Transport Protocol

The native sender uses UDP-style Sigma frames:

- Request prefix: `55 A3`
- Reply prefix: `55 A4`
- CRC: X25-style over payload, init `0xffff`, reflected poly `0x8408`, final complement
- Good response status: `0x9000`

Known text/program upload sequence:

1. `04:01` prepare
2. `02:04` upload `D\0temp.Nmg`
3. `02:0e` commit `D:\T\temp.Nmg`
4. `02:02` upload `SEQUENT.SYS`
5. `04:02` play

The native Swift `SigmaClient` now supports chunked file upload at 768-byte chunks. The old one-packet upload failed for larger bitmap/NMG files with `0x9013`.

## Editor-Compatible Send Experiment

Editor v3.99 sends a fuller `.Nmg` wrapper than LEDctrl's original lightweight sender. The vendor files start with `NGP\0` and include a post-text table plus a fuller `SEQUENT.SYS`/`SequentList.tmps` layout. LEDctrl's earlier lightweight messages displayed text correctly, but the physical sign used different spacing than the extracted `Normal7.fnt` preview. That strongly suggests the sign was taking a different firmware render path from Editor.

Current experimental single-message path (not default in live app):

- `SigmaClient.sendEditorText(...)` builds an `NGP\0` `.Nmg` payload.
- It keeps the same upload sequence: prepare, upload `D\0temp.Nmg`, commit `D:\T\temp.Nmg`, upload `SEQUENT.SYS`, play.
- The first implementation copies the post-text table template from `/Users/alexscott/Projects/MessageMaker/sigma3000_extracted/temp.Nmg`.
- It copies the Editor sequence template from `/Users/alexscott/Projects/MessageMaker/sigma3000_extracted/SequentList.tmps`.
- `RescueModel.sendCurrentMessage()` currently uses the known-good lightweight `sendText(...)` path by default.
- `sendEditorText(...)` is kept as an experiment for controlled testing and further decoding.
- Multi-row marquee programs use `sendTextProgram(...)` until Editor-style sequence generation is fully decoded.

Known template offsets from the live Editor artefacts:

- Full Editor `.Nmg`: `/Users/alexscott/Projects/MessageMaker/sigma3000_extracted/temp.Nmg`
- Full Editor sequence: `/Users/alexscott/Projects/MessageMaker/sigma3000_extracted/SequentList.tmps`
- Current `temp.Nmg` length: `5283`
- Header begins: `4E 47 50 00 07 00 F7 00 00 00 41 00 ...`
- `u16@6 = 247`, so the post-text table starts at `247 + 18 = 265`.
- `u16@10 = 65`, which appears to be visible/display length including carriage returns while ignoring control bytes.

Safety note: a previous attempt to prepend `NGP\0` into the lightweight path rendered header-like bytes on the sign. Keep `sendText(...)` as fallback and only use the fuller Editor path with the matching Editor-style sequence metadata.

## Text NMG Header

Editor-generated `temp.Nmg` files begin with:

```text
4E 47 50 00 ...
```

The message-control section includes:

- `1B 30 61`: auto-typeset / word-wrap off
- `1B 30 62`: auto-typeset / word-wrap on
- `0A 49 xx`: in effect
- `0A 4F xx`: out effect
- `0F xx`: speed
- `1C xx`: font colour / palette
- `1D 30`: background black
- `1A xx`: font size
- `07 30`: text begins after this
- `0D`: line separator
- `04`: end text before `NoteNmg file version:v3.99`

Known font/size bytes:

- `1A 30`: 5x5 / compact
- `1A 31`: 6x7 / normal 7-high
- Larger size codes exist but are not yet fully mapped to the physical 80x7 panel.

Current native font choices:

- `Compact 5-high`
- `Readable 5x7`

LEDctrl now renders canvas/preview glyphs from the vendor bitmap fonts under `/Users/alexscott/Projects/MessageMaker/sigma3000_extracted/FONT`: `Normal7.fnt` for 6x7 / 7-high text and `Normal5.fnt` for 5-high text. The parser treats each character as fixed-height rows indexed by ASCII code (`code * height`) and uses the high bits of each row byte as LED pixels. Example: `Normal7.fnt` letter `D` decodes as `####.. / .#..#. / ... / ####..`, matching the sign's rounded-D look. The old hand-made glyph table remains only as a fallback if the extracted font files are unavailable.

Uppercase and lowercase are separate glyphs in the same vendor font files. LEDctrl no longer forces canvas/preview/send text to uppercase; lowercase ASCII is allowed through the sanitizers and rendered by direct ASCII-code lookup in `Normal7.fnt` / `Normal5.fnt`.

Font spacing correction: do not add extra inter-character spacing on top of `Normal7.fnt` / `Normal5.fnt`. The vendor glyph cell already includes blank columns; LEDctrl's renderer should advance by the glyph width only. A previous `3`-pixel extra gap made `RED RED RED` visibly wider than Editor v3.99's `THIS IS RED`.

Hold / restart-gap finding:

- Captured Editor files contain `0E 32 30 30 30 32` when the UI showed `Pause 2 Secs`.
- Manual decode: `0E` is frame pause, `32` means long stop in seconds with a 4-digit value, so `0002` = 2 seconds.
- On marquee playback, the physical sign appears to leave a restart gap after text exits left before it starts again from the right.
- Native preview should model the post-exit blank gap, not snap the message back to centre. Current app preview uses at least a 1-second blank restart gap for marquee/move-left scrolling, or the selected Hold value if larger.

## Progress / Pixel Graphics

LEDctrl now has a `Progress` tab. It renders an actual 80x7 progress bar frame into RGB565 BMP data, wraps that BMP in a Sigma picture NMG, and sends a timed sequence of picture frames via `SigmaClient.sendNmg(... fileType: .picture)`. This is the current preferred route for pixel-perfect features such as progress bars, icons, custom bitmap fonts, and simple animations. It is not a high-speed live framebuffer yet; each animation frame is a file/program upload, so tune frame counts conservatively.

Mixed font-height caution: highlighted `Compact 5-high` text sent inline `{font5}` / `1A 30` is accepted by the firmware but baseline-aligns differently from LEDctrl's centred canvas. A first attempt to route mixed-font fitted rows through picture NMGs caused the sign to blank/hang during a multi-row mixed-style test, likely because the app mixed text-style row sequencing with picture NMG uploads. That fallback has been removed from Messages. Keep pixel-perfect bitmap work isolated in the Progress tab / dedicated experiments until the picture sequence format is captured and proven.

Multi-row playback correction: app-driven row-by-row uploads are wrong because each upload replaces/restarts the active sign program, causing blanking and/or the final uploaded row to repeat. LEDctrl now sends the canvas rows as one multiline message again.

NGP caution: a first attempt to emit the full Editor-style `NGP` file header through the existing `sendText` path caused the sign to render header bytes (`"` / `\"`) at the start of row 1. This means the current native `sendText` upload/sequence path expects the inner message body shape, not a complete Editor `NGP` file. Full Editor compatibility needs a separate path that sends a complete captured/generated `.Nmg` plus matching `SEQUENT.SYS`/commit semantics exactly as Editor does. Do not re-enable `NGP` in `makeNmg` until that is implemented and tested offline first.

Row gap update: to avoid separate uploads while giving visual breathing room, LEDctrl now joins canvas entries with a blank line (`\n\n`) before generating the single multiline message. This is a conservative approximation of a blank sequential entry; if the sign ignores it or wraps oddly, capture Editor sending a blank entry between two text entries and copy that exact separator/control sequence.

## 2026-05-03 Insert Picture vs Insert Back Image (wire capture)

Captured on `en0` via UDP/9520 against sign `192.168.11.6`.

### Insert picture

- Editor uploads `temp.Nmg` in two file chunks:
  - chunk 1 payload length `808`
  - chunk 2 payload length `671`
- Descriptor fields in file upload packets:
  - total size `0x0577` (`1399`)
  - max chunk `0x0300` (`768`)
  - total packets `0x0002`
  - current packet `0x0001`, then `0x0002`
- Then commit (`02:0E`), upload `SEQUENT.SYS` (`84` bytes payload), play (`04:02`).

### Insert back image (`All frame`)

- Editor enters a different payload mode and sends many chunks from one session/port.
- First observed back-image file chunk shows:
  - total size `0x249A` (`9370`)
  - max chunk `0x0300` (`768`)
  - total packets `0x000D` (`13`)
  - current packet starts at `0x0001`, then `0x0002`, `0x0003`, etc.
- The first chunk body layout differs from Insert picture:
  - starts with `01 5A 30 30 02 41 ...` but includes additional control/text fields after `RC`
  - file body appears to contain frame/background blocks, not just one picture payload.

### Practical consequence

- `Insert picture` and `Insert back image` are not equivalent protocol paths.
- A replay path that assumes one picture-style payload (`0x0577` / 2 chunks) will not faithfully reproduce back-image behavior.
- For LEDctrl parity, back-image/all-frame must be treated as a dedicated sender mode with multi-chunk program semantics (13+ chunks) and exact control block preservation.
